# CLAUDE.md — "Попутчик ИТМО"

Этот файл — инструкции для Claude Code при работе в этом репозитории.

## 0. Перед началом любой работы

**Обязательно прочитай все файлы в `ai_context/`** перед тем, как
приступать к задаче:

- `ai_context/goal.md` — цель проекта, целевая аудитория, ключевые сценарии.
- `ai_context/architecture.md` — архитектура (стек, слои, структура папок,
  схема БД).
- `ai_context/current_state.md` — что уже реализовано на данный момент.
- `ai_context/rules.md` — правила разработки.
- `ai_context/plan.md` — пошаговый план реализации и текущий шаг.
- `ai_context/design_system.md` — дизайн-система (токены, компоненты).

Эти файлы — источник истины о проекте. Если что-то в коде противоречит
им — приоритет у `ai_context/`, а расхождение нужно явно обсудить с
пользователем.

## 1. Изучай контекст перед написанием кода

Никогда не пиши код, не изучив:

- соответствующие существующие файлы (компоненты, хуки, сервисы, типы);
- текущую архитектуру (`architecture.md`) и текущее состояние
  (`current_state.md`).

Не дублируй уже существующую логику/типы/компоненты — переиспользуй их.

## 2. Архитектура — не нарушать без обновления документа

Не нарушай архитектуру, описанную в `ai_context/architecture.md`
(стек, слои UI → hooks → services → api → Supabase, структура папок,
схема БД).

Если задача требует изменить или расширить архитектуру (новая таблица,
новый слой, смена библиотеки и т.п.):

1. Сначала обнови `ai_context/architecture.md`, обосновав изменение.
2. Только затем пиши код.

## 3. После завершения каждого пункта плана

Обязательно обнови:

- `ai_context/current_state.md` — что реализовано, какие файлы/таблицы/
  эндпоинты появились;
- `ai_context/plan.md` — отметь пункт как выполненный, уточни, какой шаг
  следующий ("в работе");
- `ai_context/architecture.md` — если появились новые модули, таблицы,
  эндпоинты или изменилась структура;
- `ai_context/design_system.md` — если появились новые UI-токены/паттерны.

## 4. Дизайн-система

Используй `ai_context/design_system.md` для всех UI-компонентов: цвета,
отступы, типографику, готовые компоненты shadcn/ui. Не изобретай новые
визуальные решения без необходимости. Если дизайн-системы для нужного
случая не хватает — расширь `design_system.md` осознанно, согласовав с
пользователем при значимых решениях.

## 5. Технологический стек

Соблюдай стек, зафиксированный в `ai_context/architecture.md`:

- React + TypeScript (strict) + Vite
- Tailwind CSS + shadcn/ui
- Telegram Mini Apps SDK (`@telegram-apps/sdk-react`)
- Supabase (PostgreSQL, Auth, Storage, Realtime, RLS)

Не добавляй новые крупные зависимости/фреймворки без согласования.

## 6. Интеграции и секреты

Для любых интеграций (Telegram Bot Token, Supabase URL/anon key/service
role key и т.п.):

- ВСЕГДА запрашивай актуальные значения у пользователя;
- сохраняй их только в `.env` (локально), используя плейсхолдеры из
  `.env.example`;
- НИКОГДА не хардкодь секреты в коде;
- НИКОГДА не коммить `.env` — он должен быть в `.gitignore`.

## 7. Если не хватает информации

Если для выполнения задачи не хватает данных — доступов, токенов,
бизнес-правила, не описанного в ТЗ или `ai_context/` (например, точные
статусы из разделов ТЗ 5.3.1 и 5.6) — **остановись и спроси
пользователя**. Не додумывай и не угадывай бизнес-правила самостоятельно.

## 8. Команды проекта

```bash
npm run dev       # запуск dev-сервера Vite
npm run build     # сборка (tsc -b && vite build)
npm run lint      # ESLint
npm run preview   # предпросмотр production-сборки
```
