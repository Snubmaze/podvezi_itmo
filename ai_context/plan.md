# План реализации — "Попутчик ИТМО"

Статусы: `в работе`, `не начато`, `готово`.

После завершения каждого шага: обновить статус здесь, обновить
`current_state.md`, при необходимости — `architecture.md` /
`design_system.md`.

---

## 1. Инициализация — **готово**

Создание `ai_context/`, `CLAUDE.md`, инициализация Vite + React + TS,
подключение Tailwind + shadcn/ui + Telegram Mini Apps SDK + Supabase JS,
базовая структура папок, `.env.example`. Без бизнес-логики и экранов.

## 2. БД и Supabase — **готово**

Миграции применены к Supabase-проекту через Management API: таблицы
`users`, `locations` (единая таблица точек вместо `campuses`/`dormitories`),
`routes`, `vehicles`, `driver_documents`, `moderation_requests`, `trips`,
`trip_members`, `trip_requests`; enum-типы; RLS включён с явными политиками
на всех таблицах; приватный Storage-bucket `driver-documents` с политиками.
Справочники засидированы (5 корпусов, 5 общежитий, 50 маршрутов). Детали —
в `current_state.md` и `architecture.md` (раздел 5.2).

## 3. Telegram-авторизация, регистрация (номер ИСУ) и мок ITMO ID — **готово**

- **3a — готово**: весь UI-флоу на мок-провайдере авторизации (экраны
  Splash 6.1, регистрация ИСУ 6.2, мок-вход ITMO ID, заглушка главного
  экрана), `services/itmoId.ts`, `services/auth.ts` (интерфейс +
  мок-реализация), `AuthProvider`/`useAuth`/`AppFlow`.
- **3b — готово**: Edge Function `telegram-auth` (проверка HMAC initData +
  Admin API, задеплоена, проверена end-to-end), Supabase-реализация
  `AuthBackend`, запись `public.users`, секреты через Management API.
  Остаётся прогон клиентского флоу в Telegram на устройстве. Детали —
  `current_state.md`, стратегия — `architecture.md` 5.1.

## 4. Роли и профиль — **готово**

Экран профиля (данные из ITMO ID нередактируемые; «Описание» 5.8
редактируемое). Локальный переключатель активной роли на главном экране
(Пассажир / Водитель-заблокирован), модалка верификации по статусу
(заявка — UI-заглушка). Навигация: аватар → профиль. Безопасность: триггер
`guard_users_update` + гейт RLS `trips` по `approved` (проверено
end-to-end). Модель — `architecture.md` 5.1.1.

Полная форма заявки водителя (авто `vehicles` + загрузка документов ВУ/СТС
в Storage + `moderation_requests`) — **шаг 7**.

## 5. Маршруты — **готово**

Сервис/хук для справочника точек (`locations`), переиспользуемый
`LocationPicker`/`RouteSelector` (выбор только из справочника, корпуса +
общежития с группировкой и иконками), валидация пары «отправление —
назначение» (`validateRoutePair`). Подключено как фильтр на главном экране
пассажира. Детали — `current_state.md`, решение по `routes` —
`architecture.md` 5.2.3.

## 6. Создание и поиск поездок — **готово**

Создание поездки водителем (`TripCreateScreen`, экран 6.7), главный экран
водителя с действиями «Создать поездку»/«Мои поездки»/«Поездки без
водителя» (заглушка, экран 6.6), поиск поездок пассажиром с фильтрами по
маршруту/дате/времени (экран 6.8). Заявки на участие — через апрув
водителя: `trip_requests` (pending) → водитель принимает/отклоняет →
атомарно создаётся `trip_members` и уменьшается `seats_available` (RPC
`accept_trip_request`). «Мои поездки» — для пассажира (свои заявки) и
водителя (заявки + подтверждённые пассажиры по каждой поездке).
Realtime-подписки на `trips`/`trip_requests`/`trip_members`. Детали —
`current_state.md`, контракт сервиса — `architecture.md` 5.5.
