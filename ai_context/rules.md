# Правила разработки — "Попутчик ИТМО"

## Код и стиль

- **TypeScript strict mode** включён (`"strict": true` в `tsconfig`).
  Никакого `any` без явной необходимости и комментария-обоснования.
- **ESLint + Prettier** — обязательны для всего кода. Перед коммитом код
  должен проходить `npm run lint` без ошибок.
- Именование файлов и компонентов — по конвенциям React/TS-проекта
  (компоненты — `PascalCase.tsx`, хуки — `useSomething.ts`, утилиты/сервисы
  — `camelCase.ts`).

## Слои приложения

- **Все обращения к Supabase — только через `src/api/` (или
  `src/lib/supabase`) и `src/services/`.** Компонентам и страницам
  напрямую вызывать `supabase.from(...)`, `supabase.auth`,
  `supabase.storage`, `supabase.channel(...)` и т.п. — запрещено.
- Бизнес-логика (валидация, преобразования, композиция запросов) — в
  `src/services/`. Хуки (`src/hooks/`) — тонкая прослойка между UI и
  сервисами, отвечающая за состояние экрана/side-effects.
- Общие типы и enum'ы — в `src/types/`, синхронизированы со схемой БД
  (см. `architecture.md`).

## База данных / Supabase

- **Все таблицы Supabase — с включённым Row Level Security (RLS) и
  явными политиками доступа.** Таблица без RLS-политик в продакшен не
  попадает.
- Миграции схемы — версионируются (SQL-файлы в `supabase/migrations/`).
- Изменение схемы БД (новая таблица/колонка) — сначала отражается в
  `ai_context/architecture.md`, затем — миграция, затем — типы в
  `src/types/`.

## UI / дизайн-система

- UI-компоненты — переиспользуемые, на базе **shadcn/ui**.
- Стили — через **Tailwind CSS** и токены, описанные в
  `ai_context/design_system.md`. Не изобретать новые цвета, отступы,
  радиусы, типографику и т.п. без необходимости — сначала проверить, нет
  ли подходящего токена/компонента в дизайн-системе. Если токена не
  хватает — расширить `design_system.md` осознанно, а не точечным хаком в
  компоненте.
- Учитывать темы Telegram (light/dark, `themeParams`) — см. шаг 10 плана.

## Типизация статусов и ролей

- Все статусы и роли, явно описанные в ТЗ (в т.ч. разделы 5.3.1 — статус
  верификации водителя, и 5.6 — статус поездки), должны быть выражены как
  TypeScript `enum`/union types в `src/types/` и согласованы с
  Postgres-типами/`CHECK`-constraints в миграциях.
- Если конкретный набор значений статуса не зафиксирован ни в ТЗ, ни в
  `ai_context/`, ни в коде — **не придумывать значения самостоятельно**,
  а уточнить у пользователя.

## Персональные данные и 152-ФЗ

Проект обрабатывает персональные данные студентов: ФИО, номер ИСУ, номер
телефона, данные документов водителя (вод. удостоверение, СТС, гос. номер
автомобиля и т.п.). В соответствии с 152-ФЗ "О персональных данных":

- Доступ к персональным данным — только через RLS-политики, ограничивающие
  чтение/запись владельцем записи и администраторами/модераторами с
  соответствующей ролью.
- **Запрещено логировать персональные данные в открытом виде** (в консоль,
  в системы мониторинга, в сообщения об ошибках, отправляемые на клиент).
  При логировании ошибок — маскировать/исключать поля `full_name`,
  `isu_number`, `phone`, данные документов и т.п.
- Файлы документов водителей — хранить в приватных Storage-bucket'ах с
  доступом по RLS-политикам (владелец + модераторы), не в публичных
  bucket'ах.
- Не передавать персональные данные третьим сторонам/во внешние сервисы
  без явного согласования с пользователем.

## Секреты и переменные окружения

- Все интеграционные значения (Supabase URL/anon key/service role key,
  Telegram Bot Token, и т.п.) — **только через переменные окружения**
  (`.env`, `.env.local`), никогда не хардкодятся в коде.
- `.env` (и любые файлы с реальными секретами) — в `.gitignore`, не
  коммитятся. В репозитории хранится только `.env.example` с
  плейсхолдерами.
- Если для задачи нужен токен/ключ/доступ, которого нет в `.env` — нужно
  остановиться и запросить его у пользователя.

## Процесс работы с задачами

- Перед началом любой задачи — прочитать все файлы `ai_context/` (см.
  `CLAUDE.md`).
- Не нарушать архитектуру из `architecture.md`. Если архитектуру нужно
  изменить — сначала обновить `architecture.md` с обоснованием, потом
  писать код.
- После завершения шага — обновить `current_state.md` и `plan.md` (и
  `architecture.md`/`design_system.md`, если появились изменения).
- При нехватке информации (доступы, токены, неописанные в ТЗ
  бизнес-правила) — остановиться и спросить пользователя.
