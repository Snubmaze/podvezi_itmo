# Промты для Claude Code: ITMO Ride-Sharing Telegram Mini App

Документ построен на основе ТЗ (ГОСТ 34.602-2020) и референсов дизайна ИТМО.
Используй промты последовательно — каждый следующий выполняется только после
завершения и проверки предыдущего.

---

## Часть 0. Дизайн-система — готова

Дизайн-система спроектирована и приложена отдельным файлом `design_system.md`
(палитра light/dark, типографика, спейсинг, спецификации компонентов
Button/Badge/Card/BottomTabBar/ProgressRing/Avatar/EmptyState, иконки,
pixel-art акценты, привязка к темам Telegram). Прикладывай этот файл вместе
с Промтом 1 — Claude Code сразу положит его в `ai_context/design_system.md`.

Полноценные мокапы конкретных экранов (поиск, создание поездки, профиль и
т.д.) — необязательный следующий шаг, можно запросить отдельно по ходу
разработки на основе этих токенов и компонентов.

## Часть 1. Инициализация проекта и контекста (Claude Code)

Это **первый промт** для Claude Code. Он создаёт структуру `ai_context/`,
`CLAUDE.md`, инициализирует проект и запрашивает у тебя необходимые
секреты/доступы.

```
Мы начинаем разработку Telegram Mini App "Попутчик ИТМО" — сервиса для

корпусами и общежитиями.

ТВОЯ ЗАДАЧА НА ЭТОМ ШАГЕ — подготовить инфраструктуру проекта и контекст,

### 1. Создай структуру ai_context/

В корне проекта создай папку `ai_context/` со следующими файлами:

- `ai_context/goal.md` — цель проекта, целевая аудитория, ключевые сценарии
  (на основе предоставленного ниже описания ТЗ).
- `ai_context/architecture.md` — описание архитектуры: Frontend (React + TS +
  Vite + Tailwind + shadcn/ui + Telegram Mini Apps SDK), Backend (Supabase:
  PostgreSQL, Auth, Storage, Realtime, RLS), Deploy (Vercel). Опиши слои
  приложения (UI / hooks / services / api / types), структуру папок проекта,
  схему БД (таблицы из ТЗ: users, vehicles, trips, trip_members, routes,
  dormitories, campuses, driver_documents, moderation_requests, trip_requests),
  основные API-сущности.
- `ai_context/current_state.md` — текущее состояние проекта (на старте: пустой
  каркас, ничего не реализовано). Этот файл должен обновляться после каждого
  завершённого шага.
- `ai_context/rules.md` — правила разработки (см. ниже, раздел "Правила").
- `ai_context/plan.md` — пошаговый план реализации (перечисли все шаги из
  серии промтов: 1. Инициализация, 2. БД и Supabase, 3. Telegram-авторизация,
  регистрация (номер ИСУ) и мок ITMO ID, 4. Роли и профиль, 5. Маршруты,

  7. Регистрация и верификация водителя, 8. Админ-панель модерации, 9. Поездки
  без водителя (release 2), 10. UI/UX полировка (dark/light, Telegram themes),
  11. Тестирование и деплой, 12. Документация). Отметь текущий шаг как
  "в работе", остальные — "не начаты".
- `ai_context/design_system.md` — вставь сюда содержимое файла
  `design_system.md`, который я прикладываю к этому сообщению (готовая

### 2. Создай CLAUDE.md в корне проекта

Файл должен содержать примерно следующие инструкции (адаптируй формулировки,
суть сохрани):

- Перед началом ЛЮБОЙ работы — обязательно прочитать все файлы в `ai_context/`
  (goal.md, architecture.md, current_state.md, rules.md, plan.md,
  design_system.md). Эти файлы — источник истины о проекте.
- Никогда не писать код, не изучив соответствующие существующие файлы
  и текущую архитектуру.
- Не нарушать архитектуру, описанную в `architecture.md`. Если в рамках
  задачи архитектуру необходимо изменить/расширить — сначала обновить
  `architecture.md`, обосновав изменение, и только потом писать код.
- После завершения каждого пункта плана — обязательно обновить:
  `current_state.md` (что реализовано), `plan.md` (отметить пункт как
  выполненный, уточнить следующий), и при необходимости `architecture.md`

- Использовать дизайн-систему из `design_system.md` для всех UI-компонентов —
  не изобретать новые цвета/отступы/компоненты без необходимости.
- Соблюдать технологический стек, зафиксированный в `architecture.md`
  (React, TypeScript, Vite, Tailwind, shadcn/ui, Telegram Mini Apps SDK,
  Supabase).
- Для интеграций (Telegram Bot Token, Supabase URL/keys и т.п.) —
  ВСЕГДА запрашивать значения у пользователя и сохранять их в `.env`
  (никогда не хардкодить, никогда не коммитить `.env` в git — добавить
  в `.gitignore`).
- Если для выполнения задачи не хватает информации (доступы, токены,
  бизнес-правило, не описанное в ТЗ) — остановиться и спросить пользователя,
  а не додумывать самостоятельно.

### 3. Правила разработки (ai_context/rules.md)

Зафиксируй в этом файле:

- Стиль кода: TypeScript strict mode, ESLint + Prettier.
- Все обращения к Supabase — через выделенный слой `src/services/` или
  `src/api/`, без прямых вызовов из компонентов.
- Все таблицы Supabase — с включённым RLS и явными политиками доступа.
- Компоненты UI — переиспользуемые, на базе shadcn/ui, стили — через
  Tailwind и токены из design_system.md.
- Роуты/состояния приложения — типизированы (TypeScript enum/union types
  для статусов: верификация водителя, статус поездки и т.д., как описано
  в ТЗ разделы 5.3.1 и 5.6).
- Соответствие ФЗ-152 "О персональных данных": персональные данные (ФИО,
  номер ИСУ, документы водителя) хранить с ограниченным доступом через RLS,
  не логировать в открытом виде.

### 4. Инициализация проекта

- Инициализируй проект: Vite + React + TypeScript.
- Подключи Tailwind CSS, shadcn/ui, Telegram Mini Apps SDK (@telegram-apps/sdk
  или аналог), Supabase JS client.
- Настрой базовую структуру папок согласно architecture.md (например:
  `src/components`, `src/pages` или `src/screens`, `src/services`,
  `src/hooks`, `src/types`, `src/lib`).
- Создай `.env.example` с плейсхолдерами для переменных окружения (Supabase
  URL, Supabase anon key, Telegram Bot Token и т.д.) — без реальных значений.

### 5. Запроси у меня необходимые данные

В конце — задай мне список вопросов/запросов, которые нужны для дальнейшей
работы, в том числе:
- Токен Telegram-бота (для настройки Mini App через BotFather) и username бота.
- Создан ли у меня проект в Supabase? Если да — нужны Project URL и anon key
  (service role key — только для серверных операций, если потребуются).
- Будет ли у проекта собственный backend (Node/Express/Edge Functions) поверх
  Supabase, или вся логика реализуется через Supabase напрямую + RLS?
  (В ТЗ упомянута необходимость отдельного backend API — уточни со мной,
  как будем это реализовывать: через Supabase Edge Functions или отдельный
  сервис).
- Список фиксированных точек маршрутов (корпуса и общежития ИТМО) для
  заполнения справочников `campuses` и `dormitories` — если у меня есть
  актуальный список, я предоставлю; если нет — предложи дефолтный набор
  на основе открытых данных об ИТМО, который я потом скорректирую.

Не пиши бизнес-логику и экраны на этом шаге — только инфраструктура,
контекст и вопросы.
```

## Часть 3. Telegram-авторизация, регистрация и мок ITMO ID

```
Изучи ai_context/ перед началом работы, особенно architecture.md (схема users)
и current_state.md.

Шаг плана: "3. Telegram-авторизация, регистрация (номер ИСУ) и мок ITMO ID".

Контекст: интеграция с реальным ITMO ID/SSO (раздел 5.1 ТЗ) пока недоступна,
поэтому вместо неё реализуется ЗАГЛУШКА (мок), имитирующая поведение ITMO ID
для демонстрации сценария. В будущем заглушка должна легко заменяться на
реальную интеграцию (вынеси её за интерфейс/сервис, не смешивай с остальной
бизнес-логикой).

Задача (по ТЗ разделы 4.1, 5.1, 5.1.1, экраны 6.1, 6.2, с учётом мока ITMO ID):

1. Реализуй авторизацию через Telegram Mini Apps SDK: получение Telegram
   initData, валидацию подписи (на стороне Supabase Edge Function — опиши
   формат в architecture.md), создание/получение пользователя в таблице
   users по Telegram ID.

2. Реализуй экран начальной загрузки (6.1): логотип, название сервиса,
   индикатор авторизации.

3. Реализуй экран регистрации нового пользователя (6.2, упрощённый): только
   поле "Номер ИСУ" (числовое значение, валидация формата) и кнопка
   продолжения. Поле ФИО на этом экране НЕ запрашивается — оно будет
   заполнено автоматически на следующем шаге через мок ITMO ID.

4. Реализуй экран входа через ITMO ID (мок) — новый экран, идущий сразу
   после регистрации (для новых пользователей) или доступный из профиля
   (для повторной "привязки"):
   - Поля: "Логин" и "Пароль" (текстовые поля, без особых требований к
     формату).
   - Кнопка "Войти".
   - Поведение заглушки: ЛЮБЫЕ непустые значения логина и пароля считаются
     успешным входом (мгновенный мок-успех, без реального запроса во внешнюю
     систему). Пустые поля — показывать обычную валидацию "заполните поле".
   - При успешном "входе" вызови мок-сервис ITMO ID (реализуй как отдельную
     функцию/Edge Function `mock-itmo-id`, чтобы в будущем заменить на
     реальный запрос), который возвращает мок-профиль:
     - ФИО (например, "Полярус Павел Анатольевич")
     - курс (число, например 3)
     - возраст (число, например 20)
     - avatar_url (мок-аватар — можно использовать сервис генерации
       аватаров по строке, например на основе логина, чтобы у разных
       логинов были разные картинки)
   - Реализуй мок так, чтобы для одного и того же логина при повторном входе
     возвращались ОДИНАКОВЫЕ данные (детерминированно — например, выбор из
     небольшого набора (3-5) предопределённых "профилей" по хэшу логина),
     чтобы поведение выглядело правдоподобно при демонстрации.
   - Полученные ФИО, курс, возраст, avatar_url сохрани в таблицу users,
     установи itmo_id_linked = true.

5. После успешного мок-входа через ITMO ID — переход на главный экран
   пассажира (роль по умолчанию — пассажир). Если пользователь ранее уже
   проходил мок-вход (itmo_id_linked = true) — этот экран при повторных
   заходах в Mini App не показывается повторно.

6. Обнови экран профиля (если он уже есть/будет в шаге 4) так, чтобы поля
   ФИО, курс, возраст и аватар отображались как данные из ITMO ID и были
   НЕ редактируемые пользователем напрямую (так как в реальной системе они
   будут приходить из ITMO ID); поле "Описание" (раздел 5.8) остаётся
   редактируемым.

7. Реализуй сохранение JWT/сессии согласно правилам безопасности из ТЗ
   (раздел 7.4): HTTPS, JWT, RLS.

По завершении:
- Обнови current_state.md (реализованные экраны/потоки, явно укажи, что
  ITMO ID — это мок и где находится точка замены на реальную интеграцию).
- Отметь шаг 3 в plan.md как выполненный, шаг 4 — как "в работе".
- Обнови architecture.md: финальная схема users (поля курс, возраст,
  avatar_url, itmo_id_linked), описание мок-сервиса ITMO ID и контракт,
  по которому его предстоит заменить на реальный ITMO ID/SSO в будущем
  (раздел "Перспективы развития", п.16 ТЗ).
- Обнови миграции БД при необходимости.

Используй компоненты и стили из design_system.md для всех экранов.
```
